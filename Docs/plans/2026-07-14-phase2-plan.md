## 2. アーキテクチャ決定

### 2.1 ① 部位分割 — 位置weld+二面角しきい値クラスタリング(純 numpy、決定的)【BL-1 反映】

**前提(実コード確認済み)**: `load_mesh` は意図的に weld しない(`trimesh.load(path, process=False)` — io/mesh.py:107-118。WHY コメントが cube fixture の 24 頂点保持をまさに理由として挙げる)。glTF は UV シームで頂点を複製するため、**頂点 index ベースの面隣接では旧 UV シームが偽の部位境界に化ける**(レビュー実測: cube fixture は index 隣接だと angle=120° でも P==6)。`source_vertex` は io では恒等写像(io/mesh.py:141-142)なので weld の代用にならない。

**weld 規約(オーケストレーター裁定)**: segmentation 内部の**隣接構築専用の前段**として位置ベース weld を行う。
- `vertices (N,3) float64` の行を `np.lexsort` で辞書順ソート → **厳密ビット一致**の行グループを構成 → 各グループの代表 = **グループ内最小の元頂点 index**(決定的)。weld 写像 `w: (N,) int64`(元頂点 → 代表頂点)。
- **epsilon 不要**: glTF のシーム複製頂点は同一値のコピーであり、bit 一致で必ず束なる。許容誤差 weld は閾値選定という新たな技術判断を生むため採らない(再評価トリガー: bit 一致で束ならない実メッシュ事例が出たとき)。
- weld は**隣接導出にのみ**使う。面配列・ラベル・xatlas への入力は元 index のまま(メッシュを書き換えない=引数非破壊)。
- weld で面の corner が同一代表に潰れる面(位置的縮退面)は隣接導出上「全隣接カット」扱い(§2.6)。

**アルゴリズム**(`DihedralSegmenter`):
0. **位置 weld**(上記)→ 面の辺を weld 代表 index で表現。
1. **面隣接の構築(純 numpy)**: 各面の3辺 `(w[v_a], w[v_b])` を昇順正規化 → `np.lexsort` でグループ化 → **ちょうど2面**が共有する辺を隣接ペアとする。**3面以上が共有する辺(非多様体)は隣接に数えない(=必ずカット)** — 決定的・保守的。
2. **面法線と二面角**: numpy 外積で面法線。隣接ペア (i,j) の角 `θ = arccos(clip(n_i·n_j, -1, 1))`。零面積面が絡む隣接は常にカット(§2.6)。凸/凹は区別しない(minima-rule は Phase 3 候補として記録)。
3. **union-find**: `θ ≤ angle_deg` の隣接のみ union(純 numpy/Python の union-find、30行程度 — scipy 不要)。
4. **小部位マージ**: 面数 `< min_faces` の部位を「共有境界辺数が最大の隣接部位」へ反復マージ。**マージ先のラベルが存続**(NB-8)。決定的順序: 最小部位から、同数なら最小ラベル、マージ先 tie-break も最小ラベル。隣接部位が無い孤立成分はそのまま残す。
5. **正規化ラベリング**: 「所属最小面 index の昇順」で 0..P-1 に relabel。**出力 `(M,) int64`、0..P-1 連番、全面付与、RNG 不使用 → ビット単位で決定的**。

**初期パラメータ(test-designer が実装前に確定 — plan v3:138 の方式を踏襲)**: `angle_deg: float = 60.0` / `min_faces: int | None = None`(None = 自動 `max(2, M // 100)` を初期提案)。パラメータ検証(BL-5): `angle_deg` は有限かつ `0 < angle_deg <= 180`、`min_faces` は None または `>= 1` の int — 違反は `ValueError`。

**期待挙動の正直な記述(README/architecture.md に明記)**: 滑らかな単一ブロブでは P=1 に退化し Phase 1 と同等(優雅な劣化)。価値が出るのは (a) 複数連結成分を持つ AI 出力、(b) 鋭い折れ目を持つハードサーフェス。意味論的な部位分割は ML バックエンド(将来・ユーザープラグイン)の領分。

### 2.2 ② シーム決定 — 「部位境界=シーム」を構造で保証

Phase 2 のシーム決定は独立した工程を持たない。**部位ごとに別メッシュとして xatlas に渡す**(§2.3)ことで、チャートが部位を跨ぐことが定義上不可能になる。部位内部の追加カットは xatlas に委ねる。明示的シーム制御は Phase 3(`SeamStrategy`)。

**却下した代替案**: 部位境界で頂点複製した単一メッシュを 1 回の `add_mesh` で渡す案 — xatlas は内部で位置ベース頂点溶接(colocal)を行うため、複製境界頂点が再統合されチャートが部位を跨ぐ恐れがある(複数メッシュ方式なら定義上安全)。

### 2.3 ③④ 実現方式 — 部位ごと `add_mesh` × 1回 `generate`(単一アトラス一括パック)

`pack/part_pack.py` に internal 関数を新設:

```
_part_unwrap_and_pack(mesh: MeshData, labels: np.ndarray, *, resolution: int, padding_px: int)
    -> tuple[MeshData, np.ndarray, AtlasDims]   # faces/face_map の契約は _naive_unwrap_and_pack と同一
                                                # AtlasDims = NamedTuple(width: int, height: int)(§2.3a)
```

0. **入口検証(BL-5、fail-loud)**: `labels` は ndarray・shape `(M,)`・dtype int64・`np.unique(labels) == np.arange(P)`(0..P-1 連番)・`P >= 1`・`M >= 1` — 違反は `ValueError`(custom `SegmentationBackend` の出力もここで検疫される)。**P 上限ガード(NB-12)**: `P > 1024`(初期値、test-designer 確定)で `ValueError`(メッセージに `--seg-angle` 増・`--seg-min-faces` 増・`--granularity naive` の対処を明記 — 黙ってフォールバックしない)。
1. 部位 p ごとに `part_face_ids[p] = np.flatnonzero(labels == p)`(昇順=決定的な面来歴 provenance)、`part_vertex_ids[p] = np.unique(faces_p)` で頂点を局所 reindex したサブメッシュを作る。
2. **1つの `xatlas.Atlas` に部位ごと `add_mesh`(P回)→ `generate` 1回**(`PackOptions.resolution/padding` は Phase 1 と同じ渡し方 — xatlas_naive.py:143-146)。全部位が同一アトラス空間に正しい相対テクセル密度でパックされる。**確認済み(Codex 実測)**: 2メッシュ・resolution=64・padding=2 で atlas_count=1・共有 UV 空間が成立。
3. 部位 p ごとに `atlas[p]` から `(vmapping_p, indices_p, uvs_p)` を取得。部位単位で面数保存を検査(xatlas_naive.py:153-159 と同型の fail-loud)。**probe(§7)**: `atlas[p]` と `add_mesh` 順の対応を、部位ごと面数を変えた入力で実証(NB-1)。
4. **face 対応**: 部位内で `_build_face_map(local_faces_p, vmapping_p, indices_p)`(xatlas_naive.py:34-111 を import 再利用 — 移動しない)→ 局所 face_map を `part_face_ids[p][local_face_map]` で大域面 index に合成。**`_build_face_map` の ValueError は part_pack 側で部位 index を付与して再送出**(`raise ValueError(f"part {p}: ...") from e` — NB-6。元メッセージの `_naive_unwrap_and_pack:` 接頭辞では文脈が誤る)。
5. 連結: 頂点/UV/`source_vertex` を部位順に concatenate。**実装注意(レビュー付記)**: 局所の corner 整列済み面配列 `aligned_faces_p` は**累積新頂点数のオフセット**を加えてから連結する。`source_vertex` 合成は裁定6の契約(xatlas_naive.py:167-171)を部位経由で適用: `orig = part_vertex_ids[p][vmapping_p]` → `mesh.source_vertex[orig] or orig`。
6. **production 不変条件(fail-loud)**: (a) 大域 face_map が全単射、(b) 新メッシュの UV アイランド(新頂点共有による面の連結成分 — union-find)ごとに `labels[face_map]` が単一値、(c) アトラス寸法規約(§2.3a)。違反時 `ValueError` — 黙って部位混在アトラスを返さない。

**bake_maps 入力契約を満たす経路**: faces/face_map の戻り契約が naive と同一なので、`rebake` の既存整列コード `faces_old_aligned = mesh.faces[face_map]`(src/atlasmith/__init__.py:70-71)がそのまま適用される。**`bake/` は 1 行も変更しない**。

**重複面の扱い(BL-7 裁定 — v1 の「部位内緩和」は取り下げ)**: 正式制約は「**重複面(同一頂点集合の面)・非多様体は非対応(fail-loud)のまま**」(architecture.md:88-94 を維持)。部位が分かれた場合に限り偶発的に通ることが**あり得る**が、それを仕様として約束しない(min_faces マージが重複面を同一部位に吸収し、結局 `_build_face_map` の ValueError になり得るため)。Step 2-4 の文書更新もこの保守的文言に従う。

**却下した代替案(フォールバックとして温存)**: 部位ごと独立 `generate` → UV を `sqrt(A3D_p / AUV_p)` で密度正規化 → `add_uv_mesh` 再パック。二段 xatlas・密度正規化・二段 face 対応合成のコストで不採用。§7 の probe 破綻時のみ昇格(`add_uv_mesh`/`use_input_mesh_uvs` の実在は確認済み — 親調査)。両経路とも不可なら停止・報告。

### 2.3a アトラス寸法規約【BL-2 反映 — 新設】

**事実**: (i) Phase 1 の `_naive_unwrap_and_pack` は atlas の width/height を**一切参照していない**(xatlas_naive.py:140-159 — `atlas[0]` の3タプルのみ使用。正方形前提が暗黙)。(ii) Codex 実測: 2メッシュ・resolution=64・padding=2 で **width=90, height=43**(atlas_count=1)— 実アトラスは固定正方形ではなく、resolution を超過することもある。UV が実寸法基準で per-axis 正規化される場合、固定正方形 `texture_size` で焼くと x/y の実効テクセル密度が非等方になり、**アトラス幅 > texture_size のとき xatlas の padding 保証(アトラス画素基準)が出力テクセル基準で目減りし、ガター衝突(チャート間の滲み)が起こり得る** — padding 単一ソース同期(C9)の暗黙前提が崩れる。

**部位経路の規約(推奨案 (b) — 等方スケール+正方形出力。承認事項 D)**:
1. **probe(Step 2-2 初日)**: binding の UV 正規化規約を実測で確定する(候補: per-axis `u/width, v/height` / resolution 基準 / 生ピクセル。Phase 1 の単一メッシュ [0,1] 実測は per-axis 正規化と整合)。
2. `D = max(width, height)` とし、UV をアトラス画素空間へ戻してから一律 `1/D` でスケール(per-axis 正規化なら `u' = u·width/D, v' = v·height/D`)。チャートは正方形 `[0,1]²` の `[0, width/D]×[0, height/D]` 部分に等方密度で収まる。出力テクスチャは従来どおり正方形 `texture_size × texture_size`(公開契約を変えない)。余白はガター外(`valid_mask=False`)として扱われるだけで無害。
3. **padding 同期規約 v2**: xatlas へは従来どおり `padding_px` を渡し、bake のガター反復数は `g = min(padding_px, floor(padding_px · texture_size / D))` を rebake が計算して `bake_maps(..., padding_px=g)` に渡す(`bake_maps` のシグネチャ・実装は不変)。`texture_size ≥ D` なら g = padding_px で Phase 1 と同一関係。`D > texture_size`(アトラス超過)ではガターを縮めて衝突を防ぐ+**`rebake` が「実効密度が要求を下回った」旨を `warnings.warn` で通知**(cli.py:48-55 の警告前例と同型)。
4. **production fail-loud 検査**: atlas_count == 1(binding が公開しない場合の代替観測: 全部位チャートの UV バウンディングボックスが `[0, w/D]×[0, h/D]` 内に収まること+アイランド bbox の重なり検査をテスト側で実施 — NB-1)、`width > 0 and height > 0`。違反は `ValueError`(実寸法をメッセージに含める)。
- **代替案 (a)(承認事項 D の選択肢)**: 実寸法 `(height, width)` で焼き、出力テクスチャを非正方形にする。`bake_maps` は既に `size: tuple[int, int]` を受ける(transfer.py:74-76)ため技術的に可能で、xatlas のパッキンググリッドと 1:1 の最も忠実な案。ただし `texture_size` の意味(出力一辺)が変わり、正方形を期待する下流 DCC との相性リスクがある。
- **案 (c) は probe で棄却確認**: PackOptions に正方形強制の既知オプションは無い(bruteForce/blockAlign は別目的)— Step 2-2 probe で属性列挙により不存在を確認して記録。
- **naive 経路は変更しない**: 上記 (i) の暗黙正方形前提は Phase 1 の既知の潜在制限として Step 2-4 で architecture.md に記録するに留める(挙動不変)。

### 2.5 依存方向(architecture.md:37-41 への追補 — Step 2-4 で文書化)

`segmentation` を numpy 専用に保つのは bake と同じ「依存を増やさず境界を証明可能にする」原則(architecture.md:43-45)。weld・面隣接とも lexsort で trimesh なしに構築できる(テスト側 fixture では trimesh.creation を従来どおり使用可 — conftest.py:90,116 前例)。

**所有権/寿命/スレッド**: 変更なし(architecture.md:24-33)。全段シングルスレッド、`MeshData`/labels は値渡し・引数非破壊(weld は内部導出のみでメッシュを書き換えない)、新配列を返す。長寿命リソースなし。

**IR**: **`MeshData` にフィールドを追加しない**(決定)。labels は関数引数として明示的に受け渡す。理由: GLB に標準の面ラベル置き場がなく、provisional IR の変更は `__post_init__`(types.py:46-137)と全テストへ波及するのに対し、引数渡しで十分。

### 2.6 異常入力の終端挙動(naive 経路同等性基準)【BL-3 反映 — 新設】

正式制約(architecture.md:84-112)は不変。部位経路が naive 経路に対して**新たに黙認するものは無い**ことを次表で定義し、異常系テストを Step 2-2/2-3 の合否基準に含める:

| 入力 | naive 経路(現行) | 部位経路(Phase 2 定義) |
|---|---|---|
| `M == 0`(面なし) | xatlas 段で失敗(挙動未定義) | segmentation は `(0,)` ラベルを返すが、`_part_unwrap_and_pack` 入口検証が **明示 `ValueError`**(M>=1 要求) |
| corner の index 重複面(縮退) | `_build_face_map` が `ValueError`(xatlas_naive.py:62-66) | 同左(同関数を部位内で再利用。エラーに部位 index 付与) |
| 重複面(同一頂点集合) | `ValueError`(xatlas_naive.py:70-73) | 同左 — **緩和しない(BL-7 裁定)**。部位が分かれて偶発的に通ることはあるが約束しない |
| 零面積面(index 相異・位置縮退含む) | xatlas が面数を変えれば `ValueError`(xatlas_naive.py:153-159)、保存されれば通過(bake は `_DEGENERATE_AREA2` で no-op — transfer.py:26-29) | segmentation: 当該面の隣接を全カット(孤立部位化。min_faces マージで吸収され得る)/ pack: naive と同型の面数保存 fail-loud。**黙った面欠落は不変条件(全単射)が検出** |
| 非多様体エッジ(3面以上共有) | 正式制約で非対応(動作未保証) | segmentation は「カット扱い」で決定的に**処理はする**が、**対応は約束しない**(正式制約不変。pack 以降の挙動は naive と同等の未保証) |
| テクスチャ無し(maps 空) | 幾何+新 UV のみ書き出し(__init__.py:56-57) | 同左(部位経路の回帰テストを追加 — NB-13) |

## 3. 依存追加の判断【ユーザー承認事項 A】

**Phase 2 の新規依存: ゼロ**(pyproject.toml:7-21 を変更しない)。
- **scipy 不採用**: 必要なのは union-find(30行)と lexsort ベースの weld/隣接構築のみ。再評価トリガー: 10^6 面級で分割が数十秒を超えたとき/Phase 3 の SLIM がスパースソルバを要求したとき。
- **PartField 不採用(確定)**: LICENSE §3.3 非商用限定 = Apache-2.0 非互換(絶対規則3)。optional 依存としても搭載不可。
- **libigl 延期**: 当機で pip 導入不可(実測)。
- 以上を技術選定記録に残す(§8)。「依存追加ゼロ」も選定(=「何もしない」候補の採用)なので記録・承認対象。

## 4. 公開 API・CLI の変更案【ユーザー承認事項 B/C/D — 危険地帯2】

現行の公開面は **5関数+2公開型(`MeshData`・`BakeResult`)+CLI**(architecture.md:47-64 — NB-8 の数え直し)。Phase 2 で **5関数+4公開型+CLI** になる。変更は追加のみで、**シグネチャ互換(ただし既定動作は変更 — NB-11)**:

1. **`rebake` の拡張** — src/atlasmith/__init__.py:34-40 の現行シグネチャに keyword-only 引数を追加:
   ```python
   def rebake(input_path, output_path, *, texture_size=1024, padding_px=8,
              granularity: Literal["part", "naive"] = "part",
              segmentation: SegmentationBackend | None = None) -> None
   ```
   - `granularity="part"`(既定)= 部位分割→部位単位アイランド。`"naive"` = Phase 1 経路を温存。**未知の値は `ValueError`**(BL-5)。
   - `segmentation=None` は `DihedralSegmenter()`。`granularity="naive"` で `segmentation` 指定は `ValueError`。
2. **既定動作の変更(明示的な決定点)**: **既定 `"part"` を推奨**(ツールの存在理由・pre-1.0・利用者ゼロの今が最安の切替時機。P=1 退化時の出力は Phase 1 とほぼ同等)。**代替案(保守)**: 既定 `"naive"` 維持+オプトイン。ユーザー裁定(承認事項 B)。
3. **公開型の追加**: `SegmentationBackend` / `DihedralSegmenter` を `atlasmith` トップレベルへ re-export(承認事項 C)。
4. **CLI フラグ追加**(cli.py:16-36。`--granularity` は plan v3:185 で Phase 2 に先約済みの名前):
   - `--granularity {part,naive}`(既定は承認事項 B の裁定)
   - `--seg-angle <float>` / `--seg-min-faces <int>` — **argparse の `default=None` を番兵にし、「明示指定時のみ」`--granularity naive` との併用を `parser.error` で拒否**(BL-6。既定値経由では衝突判定しない)。
   - 既存 `--padding` / `--texture-size` は不変。
5. **承認事項 D — アトラス寸法規約(§2.3a)**: 推奨 = 案 (b) 等方スケール+正方形出力(`texture_size` 契約維持・余白は無害)。代替 = 案 (a) 実寸法非正方形出力(最忠実だが出力形状の意味が変わる)。どちらも技術的に成立するためユーザー裁定。
6. **変更しないもの**: `load_mesh`/`save_mesh`/`bake_maps`/`masked_psnr` のシグネチャ・契約、`BakeResult`、`MeshData` のフィールド、品質契約の数値規約(architecture.md:66-75)。
