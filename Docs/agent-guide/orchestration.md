# Atlasmith — オーケストレーション詳細

CLAUDE.md の「ワークフロー」節の詳細版。要点は CLAUDE.md、チェックリストと運用の細部はここ。

## 役割分担の原則

- オーケストレーター(メイン)が担う: 工程分割、設計判断、順序付け、統合、検証実行、コミット境界、最終受け入れ。
- オーケストレーター自身は計画作成・ファイル編集・自分が監督した実装のレビューを行わない。
- **実装はタスクの形で振り分ける**(→ [model-playbook.md](model-playbook.md)): 仕様が確定した実装は **Codex 直接 CLI**(指示追従・コーディング品質)、曖昧さ・横断判断が残る実装はメイン側 `implementer`(難所は上位モデルで起動)。どちらの場合もメインエージェント自身は書かない。プラグイン経由の委譲は使わない(2026-07-05)。
- **レビューは Claude+Codex のダブルチェック(必須ゲート)**: 計画・実装とも一次=メイン側の最上位モデル(実装者と別)、二次=独立した非メイン側 AI。二次は**直接 CLI**(`codex exec --sandbox read-only`、同期・ポーリング不要)で呼び、**プラグインを経由しない**(→ [codex-delegation.md](codex-delegation.md))。自己レビュー禁止。
- 各工程は独立してレビュー・検証・コミットできる最小単位にする。

## サブエージェントの起動

役割定義は `.claude/agents/`(researcher / planner / plan-reviewer / test-designer /
implementer / impl-reviewer / verifier)。
`subagent_type` に役割名を渡す。公開されていないランタイムでは `general-purpose` を起動し
「まず `.claude/agents/<役割>.md` を読み、その役割を引き受けよ」と指示し、`model` を明示する。
Codex への引き継ぎは [codex-delegation.md](codex-delegation.md) のテンプレートを使う。

## 計画に必ず含める項目

- フェーズの目的と期待される挙動変化
- 影響モジュール・公開 API・具体的なファイル(許可/禁止の書き込みパス)
- 実装方針(所有権/寿命、依存方向、スレッド前提、互換戦略)
- 走らせる検証コマンド(→ build-and-verify.md)
- リスクレベルとロールバック方針
- コミット可能になる条件

## 実装レビューで必ず見る項目

- 承認済み計画との差分(計画外の変更 = スコープ逸脱として指摘)
- プロジェクト固有の観点(構想段階版 — コード着地後に architecture.md の危険地帯と揃えて引き直す):
  依存追加・技術選定が technique-selection.md に記録されユーザー承認を得ているか /
  diff が承認された書き込みパス内に収まっているか / 型ヒント・ruff・LF+UTF-8(BOM なし)が
  守られているか(→ coding-style.md)
- 生成物・ビルドファイルが diff に混ざっていないか
- 検証コマンドの実行結果が本物の証拠か

## モデル選定

| 役割 | モデル方針 |
|---|---|
| オーケストレーター | 最上位 |
| 計画作成 / 計画レビュー / 実装レビュー / 検証 | 必ず最上位。妥協しない |
| レビュー二次 | Codex(独立エージェント) |
| 実装 | タスクの形で振り分け(→ model-playbook.md): 仕様確定 → Codex 直接 CLI / 曖昧・横断 → `implementer`(既定は下位、難所は昇格) |
| 調査・機械的作業 | 最安〜下位 |

**昇格条件(高リスク領域):**

architecture.md の危険地帯と対応(コード着地後に実態で引き直す):

- 技術選定・依存追加(`pyproject.toml`)— technique-selection.md の記録+ユーザー承認必須
- プロジェクト構造・公開 API の初期確定
- ワークフロー資産(CLAUDE.md / AGENTS.md、`.claude/`、`.codex/`)の変更 —
  ミラー同一性と Codex フック trust の再確立まで含めて完了

## モデル非依存の品質戦略

モデルのラインナップは変わる(Fable 世代の終了など)。**品質を特定モデルに固定せず、構造で担保する**:

1. **モデル指定はティア相対**: 品質役 = `model: inherit`(メインに追従)、作業役 = `sonnet`/`haiku`
   エイリアス。バージョン付きIDをリポジトリに書かない。メインセッションを常に最上位で運転する。
2. **セルフチェックの組み込み**: 各品質役の定義末尾に「Before you return (self-check)」がある。
   上位モデルは暗黙にやる自己検証を、下位モデルにも明示的に強制するための仕組み — 削らない。
3. **分解で難易度を下げる**: メインが最上位でない期間は、1タスクの認知負荷を下げる方向へ倒す —
   計画は1フェーズずつ、レビューは観点別の多視点レビュー(下記)を既定にする。
   「大きなタスクを賢いモデルで」より「小さなタスクを確実に」。
4. **エスカレーション・ラダー(スラッシング検出)**: 同じフェーズで (a) verifier の REFUTED が2回、
   (b) レビュー指摘→修正の往復が2回、(c) 実装のやり直しが2回 — のいずれかに達したら、
   そのまま回数を重ねない。上位ティアのモデルへ昇格して仕切り直すか、ユーザーに相談する。
   下位モデルで手戻りを繰り返すより、昇格して一発で通す方が総コストは低い。
5. **Codex を品質の柱として使う**: レビュー二次の Codex(reasoning effort = xhigh)は
   Claude 側最上位が細った時の独立した保険でもある。高リスク変更では Codex 二次を観点別に
   2本に増やす選択肢を持つ。

## 品質保証の追加パターン(高リスク時に使う)

- **テスト設計を実装前に挟む**: テストを伴うフェーズでは、計画承認後に `test-designer`
  (最上位)がテスト設計を作り、テスト自体の実装は `implementer` サブエージェントが行う。
  「失敗し得ないテスト」をこの段階で排除する。
- **多視点レビュー**: 高リスク領域の実装レビューは、1 回の総合レビューではなく
  観点別(正しさ / 所有権・寿命 / スレッド安全性 など)に 2〜3 の独立レビューを並列で走らせ、
  指摘を突き合わせる。観点は architecture.md の危険地帯から選ぶ。
- **プロジェクト固有のスペシャリスト**: 特定の失敗モードが繰り返されるなら専用レビュアーを
  `.claude/agents/` に追加する(実例: Sembazuru の security-reviewer / determinism-checker)。
  汎用テンプレートには含めない — 実在する失敗モードに対してだけ作る。

## ハーネスによる強制

- `enforce-codex-impl.mjs`(PreToolUse)がメインスレッドの実装ソース編集をブロック(対象拡張子はフック冒頭の `IMPL_SOURCE_RE`)。
- 一時解除: `ATLASMITH_ALLOW_DIRECT_EDIT=1`(ユーザー承認・1セッション限り)。
- `settings.local.json` はマシンローカル。別マシンは `claude-workflow-template/deploy.ps1` で再展開。
