## プロジェクト概要

Atlasmith は、AI生成3Dモデル(Tripo/Meshy/Hunyuan3D 等)の乱雑な UV を部位単位アイランドへ再編成し、既存テクスチャを新UVへ焼き直す Python 製 CLI ツール(ライセンス Apache-2.0)。
パイプラインは5段: ①部位分割(ML: PartField 予定・optional、幾何フォールバック付き)②シーム決定 ③平面展開(libigl SLIM 予定)④パッキング(xatlas)⑤テクスチャ焼き直し。①のみ ML、②〜⑤は決定的アルゴリズムのハイブリッド構成。
現状: Phase 0(雛形+GLB/glTF/OBJ 入出力)実装済み、Phase 1(⑤焼き直し)実装中、Phase 2 以降(①〜④)は未着手。
スタック確定: Python 3.11+(開発 pin 3.12)/ uv + hatchling / trimesh / numpy / xatlas / Pillow / ruff / pytest(技術選定記録: `Docs/agent-guide/technique-*.md`)。
詳細は `Docs/agent-guide/architecture.md` 参照。

### 規模トリアージ(着手前に必ず1回 — 過剰工程はコスト事故。2026-07-12 新設)

下の番号付き工程は**標準パス**であり、全タスクに課すものではない。着手前に規模を判定して
宣言し、該当パスで進む(判定に迷ったら1段上げる):

- **軽量(light)** — 局所的で可逆な小変更。目安: 触るファイル≤2・想定diff≤50行・
  危険地帯(`architecture.md`)非接触・公開API/スキーマ/所有権に変更なし。
  → `implementer` 1体に直行(researcher/planner/test-designer は起こさない。規約と
  検証コマンドはブリーフに埋め込む)→ `verifier` でゲート実行 → コミット。
  **クロスAI二次レビューは省略可**(完了報告に「軽量パス適用」と明記 — 黙って省略しない)。
- **標準(standard)** — 軽量の目安を超える非自明な変更 → 下の工程1〜8。ただし
  **クロスAI二次はフェーズ毎ではなくタスクの統合diffに1回**。
- **重量(heavy)** — 危険地帯・公開API・データ形式・並行処理・所有権に触る変更 →
  全ゲート+基幹フェーズは heavy-artillery(削除テスト/サボタージュテスト)。

**ユーザーが速度を明示した依頼(「さっと」「軽く」「すぐ」)にゲートを黙って盛らない** —
軽量パスを既定にし、上げるべき理由があるときだけ1行で示して確認する。

1. **調査** — `researcher`(read-only)。要約だけ受け取り、生の探索結果をメイン文脈に入れない。
2. **計画** — `planner`(最上位モデル)。
3. **計画レビュー** — メイン側 `plan-reviewer`(planner と別、最上位)一次 + **非メイン側 AI の二次(必須、直接 CLI)**。
4. **ユーザー承認** — コードを書く前に計画を提示(自律実行指示時は提示のみで続行)。
5. **テスト設計** — テストを伴うフェーズでは `test-designer`(最上位)が実装前に設計。
6. **実装** — メイン側の `implementer` サブエージェントへ委譲する(メイン自身は書かない。
   難所は上位モデルで起動)。**実装をパートナーAIへ移譲しない**(クロスCLI実装移譲は
   廃止 — 2026-07-12 ユーザー決定。Codex の実装力は Codex をメインとして使うことで活かす)。
   パートナーAIの役割はレビュー二次と相談のみ。弱点相殺: `Docs/agent-guide/model-playbook.md`。
7. **実装レビュー** — メイン側 `impl-reviewer`(実装者と別、最上位)一次 + クリーン文脈の**非メイン側 AI の二次(必須ゲート)**。二次は**直接 CLI**で呼ぶ — Claude メインなら `codex exec --sandbox read-only`(同期・ポーリング不要)、Codex メインなら `claude -p`。**プラグイン経由にしない**(→ `Docs/agent-guide/codex-delegation.md`)。
8. **統合・検証・コミット** — オーケストレーターが diff を検査し(宣言スコープとの照合
   `check-scope.mjs` は**統合 diff に対して1回**)、`verifier` でゲートを実行してからコミット。

**証拠を示す。断言しない。** 実行したコマンドと実際の出力・diff を貼る。
**行き詰まったら相談する。** 同一フェーズで反証・手戻りが2回続いたら、回数を重ねる前に
パートナーAI(非メイン側)へ構造化ブリーフで相談する(→ `Docs/agent-guide/codex-delegation.md` の相談プロトコル)。
手順の詳細・計画/レビューの必須チェック項目: `Docs/agent-guide/orchestration.md`。

## サブエージェント規律(子に適用されるのはこの節だけ)

1. **許可パスの内側だけを触る。** 割り当て外の変更が必要になったら、変更せずに親へ報告する。
2. **検証コマンドを実行し、実出力を返す。** 断言ではなく、実行したコマンドと出力そのもの。
3. **同一手法の失敗2回で停止し、試行の証拠を添えて親へ返す。** 3回目の同一試行はしない。
   自前の相談(ask-advisor)・自前のスコープ照合(check-scope)・5行宣言もしない —
   それらは親が統合点で行う。

## パートナーAI連携(Codex ⇄ Claude、相互ヘルプ)

実装はメイン側で完結させるが、**レビューの二次と相談は必ずパートナーAI(非メイン側)で行う**。
ベンダーの異なる視点によるダブルチェックは品質保証の柱であり、省略しない。

- **呼び出しは直接 CLI(プラグイン非依存):** Claude メイン → Bash から `codex exec --sandbox read-only ...`(同期実行なのでポーリング不要)。Codex メイン → `claude -p ... --permission-mode plan`。コマンドの詳細と雛形: `Docs/agent-guide/codex-delegation.md`。
- **既知の干渉:** Agent ツールをフックするプラグイン(context-mode 等)は、プラグイン経由の呼び出し(`codex:rescue` のサブエージェント)をブロックすることがある。直接 CLI はこの影響を受けない。
- **自己レビュー禁止:** 実装した本人に自分の変更をレビューさせない。二次レビューは常に非メイン側 AI。受け取った指摘・助言は「提案物」で、採否はメインが判断する。
- **フォールバック:** 直接 CLI の呼び出しが失敗した場合のみ、メイン側の独立二段(`impl-reviewer` 一次 + `verifier` 反証)で代替し、**パートナー二次を省略した旨を必ずユーザーへ報告する**(黙って省略しない)。

## モデル方針(モデル非依存)

品質保証(計画・テスト設計・計画レビュー・実装レビュー・検証)は **必ずその時点の最上位モデル**。
ただし **バージョン付きモデルIDをリポジトリに書かない** — ラインナップは変わる。

- サブエージェント定義: 品質役 = `model: inherit`(メインセッションと同モデル)、作業役 = `sonnet` / `haiku` エイリアス(各系列の最新に自動解決)。
- **メインセッションを常に利用可能な最上位で運転する**(`/model` で選択)。品質役は inherit で自動追従するため、モデルが世代交代してもこのファイルとエージェント定義の更新は不要。
- 例外: メインを節約モデルで走らせる場合は、品質役の呼び出し時に最上位エイリアスを明示指定する(品質を静かに下げない)。
- 調査・機械的作業は最安〜下位。難所は昇格。
- **品質はモデルの地力だけに頼らない** — 二重レビュー・反証・セルフチェック・多視点分解で構造的に担保する(→ `Docs/agent-guide/orchestration.md`「モデル非依存の品質戦略」)。

## ハーネス強制(文書ではなくフックが守る — Claude がメインのときのみ作動)

- `.claude/hooks/enforce-codex-impl.mjs`(PreToolUse)がメインスレッドの実装ソース `Edit`/`Write` を物理的にブロックする。
- 一時解除(ユーザー承認必須・1セッション限り): `ATLASMITH_ALLOW_DIRECT_EDIT=1` で再起動。
- `.claude/settings.local.json` はマシンローカル。別マシンでは `claude-workflow-template/deploy.ps1` で再展開。

`/compact` はマイルストーン境界で(計画・Done条件・未決事項を保持)。
