# モデル特性プレイブック(Claude ⇄ GPT/Codex)

役割分担の根拠となるモデル特性の記録。**全主張に証拠等級を付ける**:
【実測】= 当環境のセッション記録・トランスクリプト採掘での観測 /【外部】= 公開ベンチマーク・
一次資料(URL付き) /【ベンダー主張】= 提供元の宣伝(割引いて読む) /【ユーザー観察】= 運用者の
体感で外部裏どり未了。反例・新データが出たらこの文書を直す。
(裏どり実施: 2026-07-06。注意: ベンチは飽和・ハーネス依存が激しく、数字は「モデル+ハーネス」の
組み合わせ比較。この環境の実戦に最も近いのは Terminal-Bench 系)

**世代注記(2026-07-07 改訂): Fable はサブスクリプションでの提供を 2026-07-07 で終了。**
以後この環境の「Claude 最上位」= Opus 4.8(品質役の model:inherit は Opus に解決される)。
Fable 系の行の観測は歴史記録として残す — 世代交代しても失敗モードの型は再現しやすいため。

## 特性マトリクス(証拠付き)

### 上位 Claude(Fable 系 / Opus)
- **強み: メタ認知・全体計画・レビュー・対話**【ユーザー観察】+【外部(間接)】: 難問系
  SWE-bench Pro は Opus 4.8 が 69.2% で GPT-5.5 の 58.6% を大差でリード
  (morphllm/ITPro 集計) — 複雑・大規模文脈の問題解決力の傍証。
- **弱み: 検索を避け知識を自己完結させようとする**【実測・確定】: 本テンプレート作成時、
  「ultrathink キーワードで思考予算制御」という古い知識で助言しかけ(調査で誤りと判明)、
  さらにこのプレイブック自体を当初ユーザー観察の転記だけで書いて指摘された。
  自己完結癖は Fable 自身の行動として確認済み。
- **弱み: 長いセッションでツール呼び出し構文が崩壊することがある**【実測】: malformed invoke を
  同一セッションで7回以上連発した記録(複数プロジェクト)。粘らず引き継ぐ(fable-reasoning 参照)。
- コーディング: 「GPT より劣る」は**一概には言えない**【外部】— SWE-bench Verified は
  GPT-5.5 88.7% vs Opus 4.8 88.6% でほぼ同点、SWE-bench Pro は Opus 優位。
  ただし CLI エージェントとしての実戦(下記 Terminal-Bench)では Codex 側が優位。

### Opus / Sonnet(実務の作業役)
- **弱み: メインオーケストレーターとしての働きが不十分**【ユーザー観察・2026-07-07】:
  Sembazuru / NorvesLib で Opus をメインに据えた運用が不調で、**メインエージェントを Codex に
  移譲済み**(ユーザー判断)。Fable 引退後のメイン選定はこの実測を前提にする —
  Opus メインが不調なプロジェクトは「Codex メイン+Claude は二次レビュー/相談役(`claude -p`)」
  のデュアルメイン構成で運転する。反例(Opus メインで安定するプロジェクト)が出たらここを更新。
- **強み: 文書作成**【ユーザー観察】(外部の直接ベンチなし。反例も未観測)。
- **弱み: 指示追従が低く独断専行が出る**【実測・当環境で確定】: 採掘したインシデントに
  未承認着手・「Codexは使うので使わないとか独断を混ぜないでください」・明示規約(レイヤー依存)の
  再violation が複数記録されている。公開の指示追従ベンチの頭対頭データは**見つからず**
  (Codex 側と私の双方で調査、2026-07-06) — 当環境の実測を根拠として扱う。
- **弱み: コードが冗長・雑になりがち**【ユーザー観察】(外部裏どりなし。レビュー観点で補償)。

### GPT(Codex)
- **強み: CLI エージェントとしての実装力**【外部】: Terminal-Bench 2.1 で Codex CLI+GPT-5.5 =
  83.4% vs Claude Code+Opus 4.8 = 78.9%(tbench.ai 公開リーダーボード)。
  この環境の使い方に最も近いベンチで優位。
- **強み: 検索による知識補完**【外部(間接)+ユーザー観察】: BrowseComp で GPT-5.5 84.4% vs
  Claude Opus 4.7 79.3%(OpenAI 公表値 — 自社表につき割引きつつ、方向はユーザー観察と一致)。
- **強み: 指示追従**【ユーザー観察・条件付き実測】: 明示的な制約(出力形式・パス制限)への追従は
  当環境でも安定。**ただし無制約だとスコープ逸脱する実測あり**(頼んでいない範囲まで修正 —
  Sembazuru で記録済み。受け入れ前の `git diff --stat` が必須なのはこのため)。
- **弱み: 視野狭窄・同一手法への拘泥ループ**【実測】: Codex のスラッシング(同じ失敗の繰り返し、
  プロセス残留)が Sembazuru で記録済み。公開ベンチなし — 当環境の実測を根拠として扱う。

## 役割ルーティング(強みに仕事を合わせる)

(2026-07-07 注記: 「Claude 最上位」= Opus 4.8。Opus メイン不調の実測があるプロジェクト
(Sembazuru / NorvesLib)は Codex がメインオーケストレーターで、下表の「メイン」側工程は
Codex が担い、Claude は planner 相談(`claude -p --permission-mode plan`)と二次レビューで関与する)

| 工程 | 担当 | 根拠の等級 |
|---|---|---|
| 全体計画・アーキテクチャ判断・トレードオフ整理 | Claude 最上位(メイン / planner) | SWE-Pro 傍証+ユーザー観察 |
| 外部知識の調査(API仕様・最新版) | GPT(`codex exec`、検索込み)/ Claude researcher は検索必須指定 | BrowseComp+実測(Claude の自己完結癖) |
| **仕様が確定した実装** | **Codex 直接 CLI**(`codex exec --sandbox workspace-write`) | Terminal-Bench(実戦系で優位) |
| 曖昧・横断判断が残る実装/複雑・大規模文脈の実装 | Claude `implementer`(難所は上位モデル) | SWE-bench Pro(Opus 優位) |
| 文書作成 | Opus / Sonnet | ユーザー観察 |
| 計画レビュー(設計整合・広域影響) | Claude 最上位 | SWE-Pro 傍証+ユーザー観察 |
| 実装レビュー二次(仕様準拠・外部事実の裏取り) | GPT(`codex exec --sandbox read-only`) | BrowseComp+クロスベンダー原則 |
| 検証・反証 | Claude verifier | 反証フレーミングの運用実績 |

実装の振り分け基準: **「計画だけ読んで迷いなく書ける」なら Codex、「書きながら判断が要る・
リポジトリ横断の文脈が要る」なら implementer**。後者は SWE-bench Pro の結果とも整合する。

## 補償ルール(弱点の相互相殺)

1. **鮮度ゲート(対 Claude の自己完結癖)**: 外部の事実に依存する load-bearing な主張は記憶で
   答えず、公式ドキュメント/検索/実機プローブで裏を取る(実例: この文書の初版自体が
   裏どりなしの転記で、ユーザーに指摘された)。
2. **トンネル脱出(対 GPT の拘泥ループ)**: Codex への委譲文に必ず「同一アプローチが2回失敗したら
   停止し、試行内容と失敗を報告して戻せ(ループ禁止)」を含める。非自明なタスクでは「採用手法と
   却下した代替案1つ」を最初に報告させる。手法転換の判断は Claude 側で行う。
3. **スコープ拘束(対 GPT のスコープ逸脱・対 Opus/Sonnet の独断)**: 委譲文に許可/禁止パスを明記し、
   受け入れ前に必ず `git diff --stat`。Opus/Sonnet 側は phase-gates の要求・入口ゲートを通す。
4. **無駄コード検査(対 Opus/Sonnet の冗長コード)**: 実装レビューの必須観点に「この差分は最小か —
   不要な抽象・死にコード・過剰な防御・『ついで修正』はないか」を含める。
5. **Codex メイン時の規律装備(2026-07-07 整備)**: Codex 側にも Claude 同等のガードが存在する —
   `.codex/hooks.json`(apply_patch 実装ガード / Stop ミラーガード / SessionStart リマインダー)+
   `~/.codex/skills/` の $fable-reasoning / $phase-gates。**Codex のフックは trusted_hash 不一致で
   サイレント無効化される**ため、hooks.json を変更したら trust-hooks.mjs(展開先では
   `.codex/hooks/trust-hooks.mjs`、テンプレートでは `codex-templates/hooks/`)で
   ハッシュを再計算して `~/.codex/config.toml` の [hooks.state] を更新するのが完了条件
   (実測: 実在しないツール名 Edit|Write を見る転植 matcher が数日間死んでいた)。
6. **観測の更新**: 特性で説明できない挙動を観測したら、この文書を証拠等級付きで更新する
   (→ `/workflow-retro`)。ベンチ数字は世代交代のたびに引き直す。

## GPT-5.6 世代のモデル割当(Codex 側、2026-07-09 ユーザー指定 → 2026-07-10 実測確定)

証拠等級:【実測】リリース確認済み(codex-cli 0.144.0 の models_cache と実行ヘッダで検証)。
実 ID: `gpt-5.6-sol` / `gpt-5.6-terra` / `gpt-5.6-luna` — **綴り注意: Tera ではなく Terra**
(推測で ID を書かず実測してから記録した。鮮度ゲートの実例)。
エフォート実測: Sol=low〜ultra(既定low)/ Terra=low〜ultra(既定medium)/ Luna=low〜max(既定medium)。
`ultra` は「Maximum reasoning with automatic task delegation」(Sol/Terra のみ)。

| 役割 | グレード/エフォート | 根拠 |
|---|---|---|
| メインオーケストレーター(対話セッション) | **Sol・最高エフォート(Ultra)** | ユーザー指定。判断が最も高くつく場所に最高品質 |
| 実装者(implementer / spec-complete 委譲) | **Terra(`gpt-5.6-terra`)・高エフォート** | ユーザー指定。実装は Terra 帯、詰まれば昇格 |
| 調査者(researcher / 検索込み外部調査) | **Terra・中〜高エフォート** | 調査は量。品質は二系統クロスチェックで担保 |
| 機械的作業(整形・一括変換・軽い抽出) | **Luna** | 最安帯。判断を含まない作業のみ |
| Codex 側レビュー役 | **Sol・高エフォート以上** | 品質保証は最上位、の既存原則 |

- Advisor 構成は不変: メイン=GPT-5.6(Sol Ultra)、Advisor=Claude 最上位(`ask-advisor claude`)。
- 昇格規則(既存ラダーの適用): 反証/手戻り2回で1段上のグレード/エフォートへ。
  Terra 実装者が2回詰まったら Sol 実装者 or ask-advisor。
- **委譲時のモデル指定(実証済み)**: `codex exec -m gpt-5.6-terra ...` で呼び出し単位に
  グレードを指定できる(実行ヘッダで反映確認)。エフォートの呼び出し単位上書きは
  `-c model_reasoning_effort="high"` 形式が候補だが**未検証** — 検証まではグレード既定
  (Terra=medium)+タスクの重さで `-m` を使い分ける。
- **切替チェックリストの状況(2026-07-10)**:
  1. ✅ 実 ID・エフォート実機確認(models_cache 0.144.0+Terra 実行テスト)
  2. ✅ `~/.codex/config.toml` = `gpt-5.6-sol` / `ultra`(ユーザー側で切替済みを確認)
  3. ⏳ `.codex/agents/*.toml` のエージェント別モデル指定可否 — **未検証**(当面は `-m` フラグ委譲で代替可)
  4. ⏳ ベンチ数字の引き直し(5.6 の公表値が出たら。現行マトリクスの数字は 5.5 世代のまま)
  5. ⏳ 「Opus メイン不調 → Codex メイン移譲」観察の 5.6 での再評価(運用データ待ち)
